# Chrome DevTools MCP 脚本注入方法研究报告

> 生成时间：2026-05-14
> 目标：将 309KB 用户脚本注入到 OpenClaw Web UI（如 `http://127.0.0.1:18789/chat`）
> 环境：Chrome DevTools MCP（chrome-devtools_* 工具集）

---

## 1. 目标页面 CSP 约束（实测）

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self';              ← 关键限制
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
  img-src 'self' data: blob:;
  font-src 'self' https://fonts.gstatic.com;
  connect-src 'self' ws: wss: https://api.openai.com
```

**CSP `script-src 'self'` 意味着**：

| 方法 | CSP 是否阻止 |
|------|-------------|
| `<script src="http://...">` | ❌ 混合内容 + CSP 双重阻止 |
| `<script src="data:...">` | ❌ CSP 阻止（`data:` 不在 `script-src` 白名单） |
| `<script src="blob:...">` | ❌ CSP 阻止（`blob:` 不在 `script-src` 白名单） |
| 内联 `<script>code</script>` | ❌ CSP 阻止（无 `'unsafe-inline'`） |
| `eval(code)` | ❌ CSP 阻止（无 `'unsafe-eval'`） |
| CDP `Runtime.evaluate` | ✅ **绕过 CSP** |
| CDP `Page.addScriptToEvaluateOnNewDocument` | ✅ **绕过 CSP** |
| CDP `Fetch.enable` 拦截替换 | ✅ **绕过 CSP** |

**核心结论**：只有通过 CDP 通道（WebSocket）注入代码才能绕过 CSP。页面内所有常规脚本加载方式均被阻止。

## 2. CDP WebSocket 消息大小限制

| Chrome 版本 | 最大消息体 | 来源 |
|-------------|-----------|------|
| 旧版（~2017） | ~1 MB | [CDP Issue #24](https://github.com/ChromeDevTools/devtools-protocol/issues/24) |
| 现代版本 | 256 MB | Chromium 源码 `kSendBufferSizeForDevTools` |

**309KB 脚本评估**：
- 原始大小：309,078 字符
- JSON 序列化后（含转义）：约 350-400KB
- 加 CDP 消息包装：约 350-410KB
- **结论：远低于 1MB 限制，现代 Chrome 直接传输无问题**

## 3. 注入方法详细评估

### 方法 A：`navigate_page` 的 `initScript` 参数 ⭐⭐⭐⭐⭐

**原理**：Chrome DevTools MCP 的 `navigate_page` 工具支持 `initScript` 参数，内部调用 Puppeteer 的 `page.evaluateOnNewDocument()`，最终走 CDP `Page.addScriptToEvaluateOnNewDocument`。

**实测验证**：
```javascript
// 测试 1：initScript 基本功能
navigate_page(type="reload", initScript="window.__TEST__ = 'works';")
// 结果：✅ window.__TEST__ === 'works'

// 测试 2：initScript 绕过 CSP
// 在 script-src 'self' 的页面上，initScript 注入的代码正常执行
// 结果：✅ 完全绕过 CSP
```

**源码路径**（chrome-devtools-mcp）：
```typescript
// src/tools/pages.ts
if (request.params.initScript) {
  const {identifier} = await page.pptrPage.evaluateOnNewDocument(
    request.params.initScript,
  );
  initScriptId = identifier;
}
```

**优点**：
- ✅ 最简单的注入方式，一行调用
- ✅ 在页面脚本之前执行（`run-at document-start` 等效）
- ✅ 绕过 CSP
- ✅ 309KB 在消息大小限制内

**缺点**：
- ⚠️ 需要 reload 页面才能生效
- ⚠️ 每次导航都会重新注入（直到清除 identifier）
- ⚠️ 脚本在页面上下文中执行，可能需要等待 DOM 就绪

**适用场景**：预注入脚本，在页面加载前准备好环境

---

### 方法 B：`evaluate_script`（CDP `Runtime.evaluate`）⭐⭐⭐⭐

**原理**：通过 CDP `Runtime.evaluate` 直接在页面上下文执行 JavaScript 表达式。

**实测验证**：
```javascript
// 测试 1：基本功能
evaluate_script(function="() => { return 'hello'; }")
// 结果：✅ 返回 "hello"

// 测试 2：eval 绕过 CSP
evaluate_script(function="() => { return eval('1+1'); }")
// 结果：✅ 返回 2（CDP 上下文中 eval 可用）

// 测试 3：50KB 脚本
evaluate_script(function="() => { const p = 'x'.repeat(50000); return p.length; }")
// 结果：✅ 返回 50000
```

**优点**：
- ✅ 不需要 reload 页面
- ✅ 绕过 CSP（包括 eval）
- ✅ 即时执行，立即看到效果
- ✅ 支持异步函数（`async () => { ... }`）

**缺点**：
- ⚠️ 单次调用有大小限制（实测 50KB 可行，309KB 未测试但理论上可行）
- ⚠️ `function` 参数需要是有效的 JS 函数声明字符串
- ⚠️ 脚本内容需要 JSON 转义后嵌入函数体

**分块注入策略**：
```javascript
// 步骤 1：初始化数组
evaluate_script(function="() => { window.__CHUNKS__ = []; return 'ready'; }")

// 步骤 2：逐块推送（每块 ~10-30KB）
evaluate_script(function=`() => { window.__CHUNKS__.push("chunk_content_here"); }`)

// 步骤 3：拼接并执行
evaluate_script(function="() => { const code = window.__CHUNKS__.join(''); eval(code); }")
```

**适用场景**：在不 reload 页面的情况下注入脚本

---

### 方法 C：CDP `Fetch.enable` 拦截替换 ⭐⭐⭐⭐⭐

**原理**：启用 CDP `Fetch` 域，拦截页面发出的脚本请求，用本地内容替换响应。

**实现方式**（需直接调用 CDP，当前 MCP 工具不直接支持）：
```javascript
// 1. 启用 Fetch 拦截
cdp.send('Fetch.enable', {
  patterns: [{
    urlPattern: '*target-script*',
    resourceType: 'Script',
    requestStage: 'Response'
  }]
});

// 2. 监听拦截事件
cdp.on('Fetch.requestPaused', async ({ requestId }) => {
  const scriptContent = fs.readFileSync('./script.js', 'utf8');
  const base64Content = Buffer.from(scriptContent).toString('base64');
  await cdp.send('Fetch.fulfillRequest', {
    requestId,
    responseCode: 200,
    responseHeaders: [{ name: 'Content-Type', value: 'application/javascript' }],
    body: base64Content
  });
});
```

**优点**：
- ✅ 无大小限制（响应内容独立传输）
- ✅ 完全透明，页面不知道脚本被替换
- ✅ HTTPS 友好
- ✅ 不需要修改页面代码

**缺点**：
- ❌ 当前 Chrome DevTools MCP 不直接支持 `Fetch.enable`
- ❌ 需要直接访问 CDP session
- ⚠️ 实现复杂度较高

**适用场景**：需要替换页面已有脚本，或注入超大文件

---

### 方法 D：本地 HTTPS 服务器 ⭐⭐⭐

**原理**：启动本地 HTTPS 服务器（自签名证书），通过 `<script src="https://localhost:8443/script.js">` 加载。

**实测验证**：
```bash
# 生成自签名证书
openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem -days 365 -nodes -subj '/CN=localhost'

# 启动 HTTPS 服务器
python3 -c "
import http.server, ssl
httpd = http.server.HTTPServer(('127.0.0.1', 8443), http.server.SimpleHTTPRequestHandler)
context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
context.load_cert_chain('cert.pem', 'key.pem')
httpd.socket = context.wrap_socket(httpd.socket, server_side=True)
httpd.serve_forever()
"
```

**问题**：
- ❌ 自签名证书不被浏览器信任，`<script>` 标签加载会失败
- ❌ 即使使用 `--ignore-certificate-errors` 启动浏览器，CSP `script-src 'self'` 仍会阻止非同源脚本
- ⚠️ 需要浏览器启动参数 `--allow-insecure-localhost`

**适用场景**：可控测试环境，且页面 CSP 允许外部脚本

---

### 方法 E：Blob URL ⭐⭐

**原理**：在浏览器中创建 Blob URL，通过 `<script>` 标签加载。

**实测验证**：
```javascript
// 测试：Blob URL 加载脚本
const blob = new Blob(['window.__BLOB_TEST__ = "works";'], { type: 'text/javascript' });
const url = URL.createObjectURL(blob);
const script = document.createElement('script');
script.src = url;
document.head.appendChild(script);
// 结果：❌ script.onerror 触发，CSP 阻止 blob: 源
```

**结论**：CSP `script-src 'self'` 阻止 `blob:` URL 加载脚本。此方法在目标页面不可行。

---

### 方法 F：`data:` URL ⭐⭐

**原理**：创建 `data:text/javascript;base64,...` URL，通过 `<script>` 标签加载。

**问题**：
- ❌ CSP `script-src 'self'` 阻止 `data:` URL
- ⚠️ 309KB Base64 编码后约 412KB，URL 长度可能超浏览器限制

**结论**：与 Blob URL 相同，被 CSP 阻止。不可行。

---

### 方法 G：分块 `evaluate_script` + `eval()` 拼接 ⭐⭐⭐⭐

**原理**：将 309KB 脚本分成多个小块（10-30KB），逐块通过 `evaluate_script` 推入浏览器全局数组，最后用 `eval()` 拼接执行。

**实测验证**：
```javascript
// 步骤 1：初始化
evaluate_script(function="() => { window.__KATEX_CHUNKS__ = []; return 'ready'; }")
// 结果：✅

// 步骤 2：逐块推送（每块 ~10KB）
// 需要将脚本内容 JSON 转义后嵌入函数参数
// 结果：✅ 10KB 块可正常推送

// 步骤 3：拼接执行
evaluate_script(function="() => { eval(window.__KATEX_CHUNKS__.join('')); }")
// 结果：✅ eval 在 CDP 上下文中绕过 CSP
```

**优点**：
- ✅ 不需要 reload 页面
- ✅ 绕过 CSP
- ✅ 无大小限制（可无限分块）
- ✅ 每块消息大小可控

**缺点**：
- ⚠️ 需要多次 `evaluate_script` 调用（309KB / 10KB = ~31 次）
- ⚠️ 实现复杂度中等
- ⚠️ 需要确保分块边界不会破坏字符串转义

**适用场景**：在不 reload 页面的情况下注入大型脚本

## 5. 推荐方案

### 首选：方法 A（`initScript`）

对于 309KB 脚本，最简单有效的方式：

```
navigate_page(type="reload", initScript=<309KB脚本内容>)
```

**注意事项**：
1. 脚本内容需要作为字符串参数传入 `initScript`
2. 由于脚本 309KB，需要确保 MCP 工具能接受这么大的参数
3. 脚本在页面创建时执行，可能需要等待 DOM 就绪后再操作

### 备选：方法 G（分块 evaluate_script + eval）

如果 `initScript` 参数大小受限，使用分块策略：

1. 将脚本分成 10-30KB 的小块
2. 逐块通过 `evaluate_script` 推入 `window.__CHUNKS__` 数组
3. 最后用 `eval(window.__CHUNKS__.join(''))` 执行

### 高级：方法 C（Fetch.enable 拦截）

如果需要更复杂的注入逻辑（如替换页面已有脚本），使用 `Fetch.enable`。但当前 MCP 工具不直接支持，需要扩展 MCP 功能。
